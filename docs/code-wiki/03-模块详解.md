## 2. 入口与路由

### 2.1 `src/main.tsx`

职责：

- 从 `#root` 创建 React 根节点
- 包裹 `StrictMode`
- 引入全局样式 `index.css`

设计点评：

- 典型 Vite React 入口
- 简单清晰，无业务逻辑

### 2.2 `src/App.tsx`

职责：

- 配置 `BrowserRouter`
- 定义唯一业务路由 `/`
- 将 `/` 映射到 `Home`

设计点评：

- 当前路由结构非常轻
- 若以后扩展“第二关 / 第三关”，这里会成为路由拆分入口

## 3. 页面模块

### 3.1 `src/pages/Home.tsx`

这是当前应用最重要的业务模块。

职责：

- 根据 `step` 选择当前引导文案
- 从 `useSceneStore` 读取业务状态和动作
- 调用 `useTypewriter()` 生成逐字输出文本
- 在 `isLoading` 变化时模拟异步验证成功
- 组装页面整体布局与子组件

### 关键函数：`getTextForStep(step)`

作用：

- 根据步骤编号返回当前标题文案

当前定义：

- `1`：认知说明
- `2`：获取 DeepSeek Key 指引
- `3`：回填 API Key 并准备验证
- `4`：完成页，提示下一关方向

评价：

- 简洁直接
- 但与 `DialogPanel.tsx` 内的步骤标签定义重复，后续宜抽成统一配置

### 关键逻辑：模拟验证 `useEffect`

逻辑说明：

- 当 `isLoading` 为 `true` 时，启动 `900ms` 定时器
- 到期后调用 `handleSystemResponse('验证成功...')`
- 紧接着执行 `advanceStep()`

评价：

- 对原型演示非常合适
- 真实接入时应迁移到 service 层或异步 action 中

## 4. 组件模块

### 4.1 `src/components/DialogPanel.tsx`

这是当前最复杂的展示层组件。

职责：

- 显示步骤进度条
- 显示当前打字机文本
- 显示步骤对应的交互内容
- 提供回退按钮
- 显示状态标签与系统反馈

输入参数：

| 参数 | 说明 |
| --- | --- |
| `step` | 当前步骤 |
| `advanceStep` / `retreatStep` | 步骤推进与回退 |
| `typedText` | 当前已输出的标题文字 |
| `isTypingComplete` | 打字机是否完成 |
| `apiKey` | 用户输入值 |
| `isLoading` | 是否正在验证 |
| `isSubmitted` | 是否已提交 |
| `systemResponse` | 当前系统反馈 |
| `onApiKeyChange` | 输入更新回调 |
| `onSubmit` | 提交回调 |

### 关键内部变量

#### `canSubmit`

规则：

- 只有当 `apiKey.trim()` 非空且 `!isLoading` 时才允许提交

价值：

- 防止空提交
- 防止加载中重复点击

#### `stepItems`

定义了 4 个步骤标签：

- `1. 认知说明`
- `2. 获取钥匙`
- `3. 粘贴验证`
- `4. 第一次对话`

价值：

- 驱动顶部步骤胶囊状态展示

局限：

- 与 `Home.tsx` 中的步骤文案仍然分散维护

#### `statusLabel`

根据 `step` 和 `isSubmitted` 动态显示底部状态：

- `准备进入`
- `等待拿钥匙`
- `链路验证中`
- `等待粘贴`
- `第一关完成`

### 关键内部函数：`renderStepContent()`

这是 `DialogPanel` 的渲染核心，根据步骤输出不同内容。

#### Step 1

- 说明“这一关只有一个目标”
- 提供“开始拿钥匙”按钮

#### Step 2

- 给出获取 Key 的操作清单
- 补充“不会充值 / 不会创建 Key”的辅助提示
- 提供跳转 DeepSeek 控制台的外链
- 提供“我已经拿到了 Key”按钮

#### Step 3

- 提示用户粘贴 Key
- 展示输入框
- 支持按下 `Enter` 触发提交
- 显示“开始验证 / 连接中”按钮状态

#### Step 4

- 展示 `NEURAL LINK ESTABLISHED`
- 给出三个“下一步怎么玩”的建议

设计评价：

- 交互逻辑清晰，适合当前规模
- 但单个组件负责了大量步骤视图分支，后续会逐渐膨胀

### 4.2 `src/components/CharacterSilhouette.tsx`

职责：

- 使用多个绝对定位 `div` 绘制抽象角色剪影
- 提供赛博感边框、光晕、肩线与中心光轴

实现特点：

- 没有图片依赖
- 完全由 CSS 形状组合实现
- 适合快速调整视觉氛围

### 4.3 `src/components/Empty.tsx`

职责：

- 仅输出一个居中的 `Empty`

现状判断：

- 当前主流程未使用
- 更像脚手架遗留文件
- 可保留作占位，也可清理

## 5. Hook 模块

### 5.1 `src/hooks/useTypewriter.ts`

这是当前最关键的行为 Hook。

职责：

- 根据目标字符串逐字推进显示
- 按标点符号控制不同停顿时长
- 在切换目标文本时重置进度

### 关键常量：`BASE_DELAY`

- 默认逐字延迟为 `42ms`

### 关键函数：`resolveDelay(character)`

规则：

- 句号、问号、叹号：`240ms`
- 逗号、顿号：`120ms`
- 空格：`20ms`
- 其他字符：`42ms`

价值：

- 让中文文案打字机效果更接近自然停顿

### Hook 行为流程

1. 每次 `targetText` 变化时先调用 `resetTyping()`
2. 从索引 `0` 开始逐步切片
3. 每次推进后写入 `typedText`
4. 全部输出完成后调用 `finishTyping()`
5. 组件卸载或依赖变化时清理定时器并再次重置

实现评价：

- 逻辑清楚
- 复用性不错
- 但由于依赖 store，耦合度高于纯局部 Hook

### 5.2 `src/hooks/useTheme.ts`

职责：

- 从 `localStorage` 或系统偏好获取主题
- 将 `light` / `dark` 类名写入 `document.documentElement`
- 暴露 `toggleTheme()`

当前状态：

- 没有被 `Home` 或其他页面接入
- 文件风格仍保留分号写法，和仓库主体风格不完全一致

结论：

- 目前属于备用能力，不属于主链路

## 6. 状态管理模块

### 6.1 `src/store/useSceneStore.ts`

该文件是全局流程状态中心。

### 状态职责

- 记录当前流程步数
- 记录输入值与反馈信息
- 驱动打字机显示进度
- 驱动提交态和加载态

### 关键动作详解

#### `advanceStep()`

- 将步骤推进到下一步
- 使用 `Math.min` 防止超过 `4`

#### `retreatStep()`

- 将步骤退回上一步
- 使用 `Math.max` 防止低于 `1`

#### `setApiKey(value)`

除写入输入值外，还会：

- 重置 `isSubmitted`
- 关闭 `isLoading`
- 当输入非空时清空 `systemResponse`

这让用户重新编辑输入时，旧的提交态会被自然打断。

#### `submit()`

逻辑分两支：

- 若 `apiKey` 为空：写入错误提示 `先把 API Key 粘贴进来。`
- 若 `apiKey` 非空：进入提交态，并显示 `正在建立加密链路...`

注意：

- 这里没有真实网络请求
- 这里只是把页面推进到“准备异步完成”的状态

#### `handleSystemResponse(response)`

职责：

- 写入系统提示
- 关闭加载态
- 保持已提交状态

这个动作被 `Home.tsx` 的定时器完成逻辑调用。

#### `resetTyping()`

- 把 `typedText` 清空
- 把 `isTypingComplete` 重置为 `false`

## 7. 工具与样式模块

### 7.1 `src/lib/utils.ts`

只导出一个工具函数：`cn(...inputs)`

作用：

- 基于 `clsx` 处理条件类名
- 基于 `tailwind-merge` 去重和合并冲突的 Tailwind 类

这是典型的 Tailwind 项目通用工具。

### 7.2 `src/index.css`

职责：

- 定义全局字体
- 统一背景为纯黑
- 定义文本选中效果
- 提供 `.cyber-grid` 网格纹理
- 提供 `.scanlines` 扫描线纹理
- 定义 `.animate-cursor-blink` 光标闪烁动画

设计意义：

- 该文件奠定了整体氛围感
- 视觉风格并不完全依赖组件局部样式

## 8. 测试模块

### 8.1 `src/pages/Home.test.tsx`

当前测试聚焦两个场景：

#### 场景 1：完整主流程

覆盖内容：

- 首屏文案输出
- 从步骤 1 进入步骤 2
- 检查 DeepSeek 外链
- 进入步骤 3 并输入 Key
- 触发本地“验证成功”
- 进入步骤 4

#### 场景 2：上一步回退

覆盖内容：

- 从步骤 2 返回步骤 1
- 重新显示最初文案

测试策略特点：

- 使用 `vi.useFakeTimers()` 驱动打字机与加载定时器
- 每次测试前重置 Zustand store

### 8.2 `src/test/setup.ts`

职责：

- 引入 `@testing-library/jest-dom`

作用：

- 扩展 `toBeInTheDocument()` 等断言
