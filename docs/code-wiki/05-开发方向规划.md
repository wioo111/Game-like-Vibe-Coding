# 05. 开发方向规划

## 1. 现状判断

当前项目已经完成了一个非常清晰的“新手第一关”交互原型，但仍处于原型阶段而非产品化阶段。它的优势是体验路径短、风格统一、核心闭环明确；它的不足也同样明显：真实能力尚未接入，内容规模有限，工程边界尚未稳定。

可以把当前仓库定义为：

- 适合继续快速迭代的前端原型底座
- 适合验证“游戏化 AI 新手引导”产品方向
- 尚不适合作为完整产品直接扩展成复杂业务系统

## 2. 当前最值得保留的资产

### 2.1 交互定位清晰

项目没有把自己做成“又像官网又像教程又像 IDE”的混合体，而是坚持单目标引导，这一点非常重要。

### 2.2 氛围与文案一致

视觉风格、文案语气、步骤设计基本统一，说明产品表达方向已初步稳定。

### 2.3 状态模型简单

当前流程仍然是小规模有限状态，适合尽快抽象成可复用的关卡配置。

### 2.4 测试已经覆盖主闭环

尽管测试规模不大，但已经覆盖了当前核心路径，后续扩展时有基本安全网。

## 3. 当前主要问题

### 3.1 真实能力没有接入

当前“验证成功”完全依赖定时器模拟，用户无法真正完成 API 连通验证。

### 3.2 内容结构仍是硬编码

步骤文案、步骤标签、步骤视图分支分别散落在不同文件中，不利于扩展更多关卡。

### 3.3 页面编排层偏重

`Home.tsx` 同时处理布局、状态读取、异步模拟与组件组装，扩展后很容易膨胀。

### 3.4 存在未使用代码和依赖

`useTheme.ts`、`Empty.tsx`、`lucide-react`、`react.svg` 目前未进入业务主链路，影响仓库整洁度。

### 3.5 工程化仍较轻

没有部署规范、没有环境变量体系、没有 CI/CD、TS 严格度也偏低。

## 4. 推荐演进方向

建议把未来开发拆成四条主线并行推进。

### 4.1 主线一：把“原型验证”升级为“真实闭环”

目标：

- 用户输入 Key 后，系统能完成一次真实可验证的连通性检查

建议动作：

- 设计 `services/model.ts` 或 `services/connection.ts`
- 接入真实 API 请求或受控代理接口
- 把成功、失败、超时、余额不足、Key 无效等结果状态化
- 为错误状态补充清晰且新手友好的反馈文案

交付标志：

- 用户能够区分“复制错了”“Key 无效”“网络失败”“平台余额不足”等不同问题

### 4.2 主线二：把“固定四步”升级为“可配置关卡系统”

目标：

- 让“第一关、第二关、第三关”都能用统一结构配置

建议动作：

- 抽离 `scenes` 或 `levels` 配置文件
- 用数据结构描述步骤标题、正文、按钮、状态标签和交互类型
- 将 `DialogPanel` 中的分支渲染改为配置驱动渲染
- 将 `Home.tsx` 中的 `getTextForStep()` 收拢为统一配置源

示意数据结构：

```ts
type StepConfig = {
  id: string
  title: string
  subtitle?: string
  statusLabel: string
  contentType: 'info' | 'guide' | 'input' | 'result'
  actions: Array<{
    label: string
    kind: 'next' | 'back' | 'link' | 'submit'
    href?: string
  }>
}
```

交付标志：

- 新增一关时，主要工作变成“补配置和少量组件扩展”，而不是复制粘贴大段页面分支

### 4.3 主线三：把“页面驱动逻辑”升级为“服务边界清晰的前端架构”

目标：

- 降低 `Home.tsx` 的职责密度

建议动作：

- 将真实验证逻辑从页面中抽离到 service 或 store action
- 将步骤推进逻辑封装为更明确的流程管理器
- 视规模决定是否拆分为多个 store slice 或局部 Hook
- 将“动画状态”和“业务状态”解耦

交付标志：

- 页面层只负责组合，不直接写定时器和复杂异步逻辑

### 4.4 主线四：把“单次演示”升级为“可持续开发工程”

目标：

- 让项目具备多人协作和持续交付能力

建议动作：

- 提高 TypeScript 严格度
- 清理未使用依赖和遗留文件
- 补充更多针对流程边界的测试
- 增加部署说明和环境变量文档
- 视需要增加 CI 工作流

交付标志：

- 新成员可在短时间内完成安装、运行、理解和提交流程

## 5. 分阶段开发路线图

## Phase 1：稳定第一关

目标：

- 让当前第一关更像可用产品，而不仅是演示

建议内容：

- 接入真实校验逻辑
- 增加错误态与恢复路径
- 优化步骤 4 的完成页反馈
- 补齐空输入、重复提交、异常返回等测试

## Phase 2：引入第二关

目标：

- 从“拿到钥匙”过渡到“发出第一句话”

建议内容：

- 设计真实的新手 prompt 引导
- 拆分“问什么”“怎么问”“怎么判断回答是否可用”
- 引入第二关的数据结构和专属页面内容

## Phase 3：沉淀关卡引擎

目标：

- 将各关卡通用能力抽象出来

建议内容：

- 配置化步骤系统
- 通用状态标签、结果页、按钮区
- 场景级内容文件与埋点结构

## Phase 4：产品化

目标：

- 面向真实用户持续使用

建议内容：

- 接入用户保存能力
- 加入进度记录
- 构建多关卡路径
- 建立部署和监控体系

## 6. 具体技术建议

### 建议 1：先抽配置，再接 API

原因：

- 如果先接 API，但流程结构仍写死，后续扩展第二关时会重复重构

### 建议 2：保留 Zustand，但重新划分状态边界

原因：

- 当前规模下不需要更重的状态管理方案
- 但建议区分“流程状态”“输入状态”“动效状态”

### 建议 3：继续保持组件展示层纯净

原因：

- `DialogPanel` 当前虽然复杂，但仍基本遵守 props 输入模式
- 这是未来做配置化重构的重要基础

### 建议 4：补充失败路径测试

优先补的不是快照测试，而是以下行为测试：

- 空输入时的错误提示
- 输入变更后旧状态是否清理
- 验证中是否禁止重复提交
- 回退后再次前进是否状态正常

## 7. 文案与产品方向建议

除了技术演进，产品表达也值得同步规划：

- 保持“一次只做一件事”的新手策略，不要过早加过多说明
- 第二关建议聚焦“如何向 AI 发出第一句有效请求”
- 第三关可以进入“如何让 AI 修改你现有代码”
- 每一关都应有明确输入、反馈、完成条件，而不是只做氛围展示

## 8. 可立即执行的短期任务

以下任务可以作为下一轮开发 backlog：

1. 把步骤定义抽成统一配置文件
2. 把 `Home.tsx` 中的模拟验证逻辑抽离
3. 增加空输入与重复提交测试
4. 清理未使用依赖和遗留文件
5. 为“下一关”定义最小可玩闭环

## 9. 总结

这个仓库最有价值的地方，不是它已经完成了多少功能，而是它已经证明了一个明确的产品表达方向：把 AI Coding 新手教育做成有节奏、有情绪、有阶段反馈的游戏化体验。

后续开发应优先保护这条主线，再逐步补齐真实能力、配置化结构和工程化基础，而不是直接把它堆成普通的多页面表单系统。
